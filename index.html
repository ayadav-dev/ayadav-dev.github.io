About Me

usr@akash_yadav:~/bio

$ cat about_me.txt

Most developers learn to build upward—stacking frameworks atop abstractions. I chose the opposite path: I learned to build downward, peeling back layers until I could see the silicon breathing beneath the code.

It began innocently enough: Java's elegant objects, C++'s controlled memory. But I kept hitting invisible walls. Why does a pointer store an address? Not just how to dereference it—but why did engineers design memory this way in the first place?

So I went deeper.

I remember the first time I typed strace ./my_program on a Linux terminal and watched a cascade of system calls flood the screen—open(), read(), mmap(), brk()—each one a window into the OS negotiating with hardware on my program's behalf. I didn't just see what happened; I saw the conversation between user space and kernel space. I ran ltrace to watch libc functions mediate between my C code and the raw syscall interface. I poked /proc/[pid]/maps to visualize memory segments laid out like city districts: text, data, heap, stack—each with purpose, each with rules.

That curiosity led me to Assembly—not as a relic, but as the Rosetta Stone of computing. In HLASM, there are no metaphors. A register is a register. An instruction is a voltage pattern etched in silicon. When I write LA R1,DATA, I'm not abstracting—I'm commanding. And suddenly, everything clicked:

  • Pointers exist because CPUs address memory—it's in their DNA.
  • static exists because storage duration matters when every byte has physical cost.
  • JCL isn't "old"—it's a precise contract between human intent and batch-processing reality.

Today, as a Technical Lead at Fiserv, this mindset isn't academic—it's operational. When a Pro*C transaction stalls under load, I don't just profile SQL. I trace it through Oracle's precompiler, watch how embedded C binds to OCI libraries, inspect memory alignment in the generated C stubs, and correlate wait events with mainframe dispatching behavior. I see the entire stack—not as separate layers, but as one continuous chain of cause and effect.

Why does this matter to financial systems?

Because when you're processing $2B in daily transactions, abstractions break. Memory leaks become outages. Misaligned cache lines become latency spikes. And "it works on my machine" becomes a regulatory incident. My value isn't just writing Pro*C or Assembly—it's understanding why the system behaves as it does so I can architect solutions that don't just work today, but remain stable, auditable, and performant for decades.

I don't chase shiny tools. I chase first principles. And in an industry where failure means real money lost and trust broken—that depth isn't optional. It's the foundation.
